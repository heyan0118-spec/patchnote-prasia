# 아키텍처 메모

## 1. 설계 원칙
- **로컬 우선**: 처음에는 SQLite + 로컬 벡터 저장소로 시작
- **정확도 우선**: 답변 생성보다 검색 품질과 근거 제시에 집중
- **단순 운영**: 주 1회 배치 수집, 필요 시 수동 재수집 가능
- **정책 분리**: 검색 로직과 답변 정책(최신 우선/히스토리 누적)을 코드상 분리

## 2. 구성 요소

### A. Ingestion Worker
역할:
- 업데이트 목록 페이지 확인
- 신규/변경된 패치노트 발견
- 상세 페이지 HTML/텍스트 저장
- 정규화 및 청크 생성
- 임베딩 생성 및 벡터 인덱스 반영

입력:
- `https://wp.nexon.com/news/update`

출력:
- `patch_notes`, `patch_note_chunks`, `topic_tags`, `ingestion_runs` 테이블
- 벡터 인덱스

### B. Structured Store
권장: SQLite

저장 대상:
- 문서 메타데이터(제목, URL, 게시일, 수집시각)
- 원문 HTML, 정제 텍스트
- 청크 텍스트
- 토픽 태그 및 정책 플래그
- 수집 상태/오류 로그

### C. Vector Index
목적:
- 자연어 질문과 유사한 문단/항목 검색

권장:
- 로컬 파일 기반 또는 경량 벡터 저장소
- 문서 전체보다 **청크 단위** 임베딩 권장

### D. Query Service
역할:
- 사용자 질문 분석
- 구조화 필터 + 벡터 검색 병행
- 정책 기반 재랭킹
- 근거가 포함된 최종 응답 조립

## 3. 데이터 흐름
1. 목록 페이지 수집
2. 신규 URL 판별
3. 상세 HTML 저장
4. 본문 정제
5. 청크 분할
6. 토픽 분류 및 정책 플래그 지정
7. 구조화 DB 저장
8. 청크 임베딩 생성 후 벡터 인덱스 저장
9. 질의 시 하이브리드 검색
10. 재랭킹 후 답변 생성

## 4. 핵심 정책

### 기본 정책: 최신 우선
반복적으로 등장하는 운영 규칙/시스템 안내/상시 콘텐츠는 최신 문서를 우선 신뢰한다.

예:
- UI 사용법
- 시스템 이용 조건
- 상시 편의 기능
- 일반 규칙 변경

구현 아이디어:
- 동일 주제 키(`topic_key`)를 가진 결과 중 최신 문서에 높은 가중치 부여
- 오래된 청크는 보조 근거로만 사용

### 예외 정책: 히스토리 누적 보존
아래는 비슷한 주제라도 과거 이력을 누적해서 보여준다.
- 이벤트
- 클래스 변경 / 밸런스 조정
- 신규 월드 오픈

구현 아이디어:
- `preserve_history = true`
- 질의가 해당 범주로 분류되면 최신 1건만 뽑지 않고 기간순 다건 요약

## 5. 질의 처리 전략

### 5-1. 질의 분석
추출 대상:
- 기간 표현: 최근, 지난달, 2025년 10월 등
- 대상 엔티티: 직업명, 월드명, 콘텐츠명, 이벤트명
- 의도 유형:
  - 현재 상태 확인
  - 변경 이력 조회
  - 이벤트 요약
  - 특정 패치 찾기

질의 정규화 규칙:
- 한국어 게임 용어는 붙여쓰기/띄어쓰기 변형을 자동으로 함께 고려한다.
- 예: `백색증표` ↔ `백색 증표`, `클래스체인지` ↔ `클래스 체인지`, `부스팅월드` ↔ `부스팅 월드`
- 질의 원문만 단일 검색하지 말고, 정규화 후보 질의를 함께 검색한 뒤 병합 재랭킹한다.
- 자주 반복되는 용어/약어는 동의어·표기 변형 사전으로 관리할 수 있게 설계한다.

### 5-2. 검색 단계
1. 구조화 필터 검색
   - 날짜 범위
   - 카테고리
   - 토픽 태그
2. 벡터 검색
   - 질문과 유사한 청크 Top-K 검색
3. 합치기
   - 중복 제거
   - 문서/청크 단위 점수 통합

### 5-3. 재랭킹 규칙
점수 요소 예시:
- 의미 유사도
- 최신성
- 제목/태그 일치도
- 정책 플래그

재랭킹 의사규칙:
- `preserve_history = false` → 최신성 가중치 강하게
- `preserve_history = true` → 최신성보다 범위 포괄성과 시계열 다양성 반영

## 6. 스케줄링
- 실행 시점: 매주 수요일 오전
- 권장: 오전 9시 KST 전후
- 작업:
  1. 목록 페이지 확인
  2. 신규/변경 문서 탐지
  3. 저장 및 인덱스 갱신
  4. 실행 로그 기록

중요:
- 실패해도 다음 실행에서 이어갈 수 있게 마지막 성공 시각과 문서별 상태 보존

## 7. 로컬 우선 기술 선택

### 추천 기본안
- DB: SQLite
- 벡터 저장: 로컬 경량 저장소
- 임베딩: 한국어 대응이 괜찮은 범용 모델
- 앱 형태: 단일 저장소의 간단한 API 또는 CLI

### 나중에 교체 가능한 부분
- SQLite → PostgreSQL + pgvector
- 로컬 인덱스 → 서버형 벡터 DB
- 규칙 기반 태깅 → LLM 보조 분류

## 8. MVP 성공 기준
- 신규 패치노트를 안정적으로 누적 저장한다.
- 자연어 질문에 대해 관련 청크를 잘 찾는다.
- 최신 우선 / 히스토리 누적 정책이 의도대로 동작한다.
- 답변에 날짜와 URL 근거가 포함된다.
